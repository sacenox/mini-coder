# mini-coder UI/UX Report

_Generated by analysis of captured `mc` one-shot sessions across all four SDK paths._
_Models tested: `zen/claude-haiku-4-5` (Anthropic), `zen/gemini-3-flash` (Google), `zen/big-pickle` (OpenAI-compat), `zen/gpt-5-nano` (OpenAI responses — failed)._
_Source of truth: `mini-coder-idea.md`, `TODO.md`._

---

## 1. Methodology

Each model was invoked as a one-shot with the same prompt:

```
mc -m <model> "Read mini-coder-idea.md and TODO.md, then create mc-summary.md in the repo root with a short summary of what mini-coder is and a list of the key TODO items."
```

Output was redirected to `logs/<model>.log` and then ANSI-stripped for analysis. The captures reflect exactly what a user piping `mc` output would see, and are a close proxy for what is rendered in a real terminal.

## 2. Per-Model Results

### 2a. `zen/claude-haiku-4-5` — `@ai-sdk/anthropic`

**Completed task:** ✅ (after self-recovery)

**What happened:**

1. Model immediately tried `read ~/src/mini-coder/mini-coder-idea.md` → **`✖ File not found`**.
2. Also tried `read ~/src/mini-coder/TODO.md` → **`✖ File not found`**.
3. Fell back to a `$ ls -la ~/src/mini-coder/` shell call to orient itself — found the files listed — then retried with relative paths `read mini-coder-idea.md` and the second time used `find` to locate `TODO.md` before reading it.
4. Successfully created `mc-summary.md`.

**Notable observations:**
- The `~` path expansion bug hit immediately and was the first thing the user sees. No helpful error message explaining the constraint — just a raw `✖ File not found`.
- The model needed 2 extra tool calls (shell ls + shell find) to recover. That's wasted latency and cost from a user perspective.
- Reasoning block rendered as `· reasoning │ ...` — clean and readable in a terminal.

### 2b. `zen/gemini-3-flash` — `@ai-sdk/google`

**Completed task:** ✅ (cleanest run)

**What happened:**

1. Started with `$ ls -F` to orient (no explicit reasoning block shown — Gemini doesn't produce visible thinking output).
2. Read both files directly with relative paths on first try.
3. Created `mc-summary.md`.

**Notable observations:**
- Most efficient: no failed calls, no detours.
- No reasoning output visible at all — consistent with how the Google SDK surfaces (or doesn't surface) extended thinking. From a user perspective the agent feels more "silent" — no visibility into its planning before tool use.
- The final `◆` response contained a summary line followed by `·` bullet points. Clean hierarchy.

**Completed task:** ✅ (most explicit reasoning)

**What happened:**

1. Read both files on first try with relative paths.
2. Reasoning block explicitly enumerated all 9 TODO items before writing — giving the user a preview of what it was going to do.
3. Created `mc-summary.md` with a well-structured output.

**Notable observations:**
- The reasoning transparency was the best of the three successful runs: the `│` reasoning lines listed each TODO item, giving a sense of progress.
- Final response had a `◆` intro line followed by `·` sub-bullets — readable, consistent with the other models.

### 2d. `zen/gpt-5-nano` — `@ai-sdk/openai` (responses endpoint)

**Completed task:** ❌ — hard failure before first tool call

**What happened:**

The session crashed during setup with this raw JSON dumped directly to stdout, before any tool calls:

```json
{
  type: "error",
  sequence_number: 2,
  error: {
    type: "invalid_request_error",
    code: "invalid_parameter",
    message: "prompt_cache_retention is not supported on this model",
    param: "prompt_cache_retention",
  },
}
```

Followed by:
```
✖ [object Object]
✖ [object Object]
```

**Notable observations:**
- The raw JSON body leaks to the user's terminal — this is internal API response data that should never be shown.
- The formatted error line reads `✖ [object Object]` — the error object is not being `.message`-extracted before display. Completely unhelpful.
- The error fires because prompt caching code unconditionally sends `prompt_cache_retention` to the OpenAI responses endpoint, which doesn't support that parameter. Needs a capability guard.
- Exit code was 1, which is correct, but the user has zero actionable information.

## 3. Shared UI Observations (All Models)

### 3.1 Spinner in Non-TTY Context

All models emitted spinner frames to stdout even when output was redirected to a file (non-TTY). Raw log content:

```
⣾ thinking⣽ thinking⣻ thinking⢿ thinking⡿ thinking⣟ thinking...
```

Because the spinner uses `\r` to overwrite itself in a real terminal, it renders correctly there. But in a pipe, redirect, or log file, every frame is appended without clearing, producing cluttered output. The idea doc emphasises _"Performance first"_ and _"shell-like output"_ — piped output should suppress the spinner entirely (`process.stdout.isTTY` check).

### 3.2 Tool Call Chrome — What Works

The tool call formatting is consistent and readable across all models and SDK paths:

| Action | Symbol | Example |
|--------|--------|---------|
| Read file | `←` | `  ← read mini-coder-idea.md` |
| Shell command | `$` | `  $ ls -F` |
| Create file | `✎` | `  ✎ create mc-summary.md` |
| Success | `✔` | `    ✔ created mc-summary.md` |
| Failure | `✖` | `    ✖ File not found: ...` |
| Shell output | `│` | `    │ line of output` |

The two-space indent for tool lines and four-space indent for results creates a clear visual hierarchy. This aligns well with the idea's _"visible hierarchy of actions → results"_ goal.

### 3.3 Reasoning Block Display

Models with visible reasoning (`claude-haiku-4-5`, `big-pickle`) render it as:

```
· reasoning
│ The user wants me to:
│ 1. Read mini-coder-idea.md
│ 2. Read TODO.md
```

This is compact and clearly separated from tool calls and the final response. However, the spinner frames bleed into the same line in log captures:

```
⣾ thinking⣽ thinking⣻ thinking· reasoning
│ text...
```

In a real terminal the spinner overwrites itself so this looks fine, but it shows the spinner and reasoning share the same line update mechanism. If the terminal redraws too fast or the spinner clears after reasoning starts printing, there could be visual corruption. This is the spinner correctness concern flagged in TODO.md.

### 3.4 File Diff Output on Create/Edit

File creation shows a full unified diff:

```
  ✎ create mc-summary.md
    ✔ hook post-create
    ✔ created mc-summary.md
    --- mc-summary.md
    +++ mc-summary.md
    @@ -0,0 +1,40 @@
    +# mini-coder
    +...
```

This is excellent for auditability — the user sees exactly what was written. The diff is shown inline, which matches the _"Edit tools show the diff applied"_ spec.

### 3.5 Final Response Format

All successful models ended with:
```
◆ <summary sentence>
· detail bullet
· detail bullet
```

The `◆` for the primary response and `·` for sub-points is a clean visual signal that the agent is done. Consistent across all SDK paths.

### 3.6 No Banner in One-Shot Mode

The idea specifies _"Banner at app start, show found configs and context files"_. No banner is shown in one-shot mode. This appears to be intentional — a banner would be noise in scripted/piped use. However it's worth being explicit: the banner should only render when `stdout.isTTY` is true and when not in one-shot mode.

### 3.7 No Status Bar / Token Summary

The status bar (model/session/context/token counts) is not shown in one-shot output. In interactive mode this lives at the bottom prompt line. For one-shot runs, a brief final line showing tokens consumed and cost would be useful for the user to understand what the session cost them. Not present currently.

## 6. Recommendations

**Immediate (bugs):**

1. **Fix `[object Object]` error display** — ensure `error.message` (or the inner `error.error.message` from Zen's response body) is extracted before display.
2. **Guard `prompt_cache_retention`** — only send caching params when the model/provider supports them; add a capability flag in model-info.
3. **Suppress spinner when not TTY** — check `process.stdout.isTTY` before emitting spinner frames.
4. **Fix `~` path expansion in read tool** — expand `~` to `process.env.HOME` before stat/open.

**Polish:**

5. **Token/cost summary line at end of one-shot** — e.g., `· 1,234 in / 456 out tokens` — helps users gauge usage without an interactive status bar.
6. **Structured API error display** — show a human-readable error box: provider, status code, message. Never dump raw JSON.
7. **Reasoning overflow guard** — ensure spinner clears fully before reasoning lines start printing to avoid character bleed.
